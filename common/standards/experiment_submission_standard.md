# 三项目实验提交规范

## 1. 文档目的

本规范适用于南京航空航天大学与 ACOINFO 项目制实习物联网方向的三个项目及 16 个实验，用于统一实验目录、代码、文档、测试、日志、截图和 Git 提交流程。

所有提交应满足以下基本目标：

- 内容真实，由对应实验负责人完成并确认。
- 文件位置统一，其他成员能够快速定位。
- 操作步骤完整，非作者能够按照说明复现实验。
- 测试过程有依据，测试结果有记录，失败问题可追踪。
- 不提交密码、密钥、个人隐私、临时文件和无关生成文件。

## 2. 适用项目与实验分组

| 项目目录 | 项目方向 | 负责实验 |
| --- | --- | --- |
| `01_protocol_bridge/` | MQTT 与 VSOA 协议桥接 | L5、L6、Z6 |
| `02_lora_lorawan/` | LoRa/LoRaWAN 应用案例 | L1-L4、L7-L8 |
| `03_zigbee/` | ZigBee 应用案例 | Z1-Z5、Z7-Z8 |

`01_protocol_bridge/experiments/B00_common_framework/` 是公共框架目录，不计入 16 个正式实验。

## 3. 提交责任

| 角色 | 主要责任 |
| --- | --- |
| 实验负责人 | 完成实验代码、README、配置、测试、日志和结果材料，并保证内容真实可运行 |
| 项目组长 | 审核本项目实验目录、命名、完成度和组内联调结果 |
| 公共模板与测试负责人 | 维护统一模板、提交规范、测试脚本框架和公共验收格式，不代替实验负责人编写正式业务代码 |
| 交叉测试成员 | 按 README 独立复现，填写测试结果和发现的问题 |
| 指导老师或指定审核人 | 确认正式测试框架、验收结果和最终版本 |

AI 工具可以用于解释、排版、检查和辅助编写，但不能将未经负责人理解、运行和验证的 AI 生成内容作为正式实验成果。使用 AI 辅助完成的代码或文档，提交人仍对其正确性负责。

## 4. 实验目录结构

每个实验必须使用仓库中已经建立的对应目录，不得另建重复实验目录。

```text
experiments/{实验编号}_{实验名称}/
  README.md              实验入口说明，必交
  src/                   正式源代码或完整工程，必交
  config/                配置示例和参数说明，按实验需要提交
  docs/                  实验指导书、设计说明、测试文档，必交
  slides/                汇报 PPT，阶段或最终验收时必交
  tests/                 测试脚本、测试数据和测试记录，必交
    logs/                编译、烧录、串口、通信或联调日志，必交
  assets/                接线图、流程图、运行截图和结果图片，必交
```

项目级公共代码、配置或脚本可以放在项目根目录的 `src/`、`config/`、`scripts/`、`tests/` 中；仅服务于某个实验的内容必须放入对应实验目录。

空目录使用 `.gitkeep` 保留。正式文件加入后可保留或删除 `.gitkeep`，但不得删除标准目录。

## 5. 每个实验必交内容

| 序号 | 交付物 | 最低要求 |
| --- | --- | --- |
| 1 | `README.md` | 说明目标、负责人、环境、接线、配置、运行、测试和结果 |
| 2 | 正式源代码 | 能编译或运行，入口明确，不含无关代码 |
| 3 | 配置材料 | 提供配置示例、参数含义和修改位置，不含真实密码 |
| 4 | 实验指导书 | 包含原理、环境、步骤、预期结果、常见问题和复现方法 |
| 5 | 测试脚本或测试步骤 | 覆盖环境、核心功能、异常情况和结果判断 |
| 6 | 测试记录 | 写明输入、预期结果、实际结果、状态、执行人和日期 |
| 7 | 日志 | 至少保留一份能够证明成功运行的关键日志 |
| 8 | 图片证据 | 包含关键配置、运行过程和最终结果，不仅是代码截图 |
| 9 | 汇报材料 | 使用统一 PPT 模板，内容与实际实验结果一致 |
| 10 | 验收结论 | 明确通过、不通过或待复测，不得只写“已完成” |

如果实验不需要某一类文件，应在 README 中说明“不适用”及原因，不得用空白文件冒充交付物。

## 6. README 编写要求

实验 README 是复现入口，至少包含以下内容：

1. 实验名称、编号、所属项目、负责人和当前状态。
2. 实验目的、功能范围及最终现象。
3. 硬件清单、软件环境、SDK、工具和版本。
4. 接线方式、网络拓扑或模块连接关系。
5. 获取代码、导入工程、安装依赖和编译方法。
6. 配置参数、设备地址、串口参数、MQTT topic 或协议接口。
7. 烧录、启动、运行和停止步骤。
8. 测试方法、预期结果及结果文件位置。
9. 已知问题、排查方法和安全注意事项。
10. 本实验交付物清单和完成状态。

命令必须能够直接执行；需要替换的内容使用 `[参数名]` 标注，不得把个人电脑的绝对路径写成通用命令。

## 7. 源代码与工程要求

- 正式代码由对应实验负责人或开发成员提交并确认。
- C/C++、Java、Python 和嵌入式 IDE 工程均可提交，不强制实验业务代码使用 Python。
- 源代码应有明确入口，文件命名能够反映功能。
- 复杂协议转换、硬件初始化和异常处理处应有必要注释。
- 不得提交无法解释、未经运行或仅用于展示的占位业务代码。
- 不得提交编译产物、缓存、虚拟环境和 IDE 用户状态，例如 `__pycache__/`、`.venv/`、`build/`、`Debug/`、`.pytest_cache/`。
- 需要保留的固件文件应在 README 中注明来源、版本、生成方法和适用硬件。
- 第三方代码应注明来源、许可证和所做修改，不得删除原版权说明。

## 8. 配置与敏感信息要求

配置文件放入 `config/`，推荐提供 `config.example.*` 或使用明显的占位值。

不得提交以下内容：

- GitHub Token、API Key、MQTT 密码和服务器私钥。
- 校园账号、邮箱密码、设备管理密码和个人隐私信息。
- 仅在个人电脑有效的绝对路径、临时端口和未说明的局域网地址。

必须在 README 中说明：

- 配置文件复制或创建方式。
- 每个必填参数的含义和示例。
- 默认值、有效范围及修改后是否需要重启。
- MQTT broker、topic、串口、波特率、频率、节点地址等关键参数。

## 9. 测试提交要求

三个项目采用“统一测试入口和记录格式，实验内容分别补充”的方式。统一 Python/pytest 模板用于组织和调用测试，不限制正式实验代码语言。

每个实验至少覆盖：

| 测试类别 | 内容 |
| --- | --- |
| 环境检查 | 工具、依赖、硬件连接和配置是否满足要求 |
| 构建或启动测试 | 工程能否编译、烧录、启动或连接 |
| 核心功能测试 | 实验要求的主要功能是否正确实现 |
| 数据或协议测试 | 输入、输出、帧格式、topic、字段或接口是否正确 |
| 异常测试 | 非法输入、断网、超时、丢包、设备离线等情况 |
| 日志测试 | 关键过程、错误和结果是否能够定位 |
| 复现测试 | 非作者能否按照 README 独立完成 |

测试脚本命名建议：

```text
test_{实验编号小写}_{功能}.py
test_{功能}.c
{功能}Test.java
run_tests.ps1
```

Python 统一模板位于 `common/templates/test_scripts/experiment_test_template.py`。C、C++、Java 或嵌入式实验可以由该模板调用编译命令、可执行程序或串口采集程序，也可以提交本语言测试；最终仍需使用统一测试记录表汇总结果。

## 10. 测试记录要求

测试记录放在对应实验的 `tests/` 中，文件名统一为：

```text
test_record_YYYY-MM-DD.md
```

每条测试用例至少记录：

- 用例编号和测试名称。
- 测试目的、前置条件和执行环境。
- 输入数据、配置参数和操作步骤。
- 预期结果和实际结果。
- 状态：通过、不通过或待复测。
- 执行人、执行时间和证据文件路径。
- 失败原因、处理方法和复测结果。

不得在尚未执行测试时提前填写“通过”。自动测试结果不能替代硬件现象、串口输出和实际联调记录。

## 11. 日志与截图要求

实验日志放入 `tests/logs/`，项目汇总日志放入项目根目录 `logs/`。

建议命名：

```text
build_YYYYMMDD_HHMM.log
flash_YYYYMMDD_HHMM.log
serial_YYYYMMDD_HHMM.log
mqtt_YYYYMMDD_HHMM.log
integration_YYYYMMDD_HHMM.log
error_YYYYMMDD_HHMM.log
```

日志应包含时间、模块、级别、关键输入、执行结果和错误原因。日志中涉及密码、Token 或个人信息时必须脱敏。

图片放入 `assets/`，建议命名：

```text
fig01_hardware_wiring.png
fig02_build_success.png
fig03_runtime_result.png
fig04_test_result.png
```

截图要求画面清晰、裁剪合理，并在 README 或实验指导书中引用和说明。不得使用与当前版本无关的旧截图，也不得只提交聊天记录作为实验结果。

## 12. 三个项目的专项要求

### 12.1 协议桥接项目

- 明确 MQTT broker、topic 规则、QoS、payload 示例及字段含义。
- 明确 VSOA 服务、URL、数据类型、请求响应和错误处理方式。
- 分别验证 MQTT 到 VSOA、VSOA 到 MQTT及双向联调。
- 验证非法 JSON、缺失字段、未知设备、连接失败和超时。
- 正式桥接业务代码由协议桥接开发成员提交；公共模板负责人只维护测试与文档公共能力。

### 12.2 LoRa/LoRaWAN 项目

- 记录开发板、模块、频率、带宽、扩频因子、编码率和发射功率。
- 提供接线图、固件烧录方法、节点角色和网关配置。
- 测试收发、传感采集、ACK/重传、链路质量和低功耗场景。
- 涉及 MQTT 时记录网关到 broker 的 topic 与 payload。

### 12.3 ZigBee 项目

- 记录芯片或开发板型号、协议栈版本、信道、PAN ID 和节点角色。
- 提供协调器、路由器、终端节点的烧录和组网步骤。
- 测试入网、掉线重连、数据上报、下行控制和多节点场景。
- 涉及 MQTT 或 RS485 时记录接口参数、数据格式和透传结果。

## 13. PPT 与实验指导书要求

实验指导书使用 `common/templates/实验指导书.docx`，PPT 使用 `common/templates/实验汇报PPT模板.pptx`。

实验指导书应包含封面、目录、实验目的、原理、环境、准备工作、操作步骤、测试与结果、故障排查、总结和参考资料。表格保持白底黑框、字体为黑色。

PPT 至少包含项目背景、实验目标、总体方案、关键实现、测试过程、结果证据、问题与解决方法、分工和总结。PPT 中的数据、截图和完成状态必须与仓库当前版本一致。

## 14. Git 与协作流程

推荐成员通过 Fork 和 Pull Request 协作：

1. Fork 主仓库到个人 GitHub。
2. Clone 个人 Fork，并添加主仓库为 `upstream`。
3. 每个实验或任务建立独立分支，例如 `feature/L01-sensor-uplink`。
4. 只修改本人负责的实验目录或已明确分配的公共文件。
5. 本地完成运行、测试和文档检查后提交。
6. 推送个人分支并创建 Pull Request。
7. 项目组长或指定审核人检查后合并到 `main`。

推荐提交信息：

```text
feat(L01): add LoRa hello communication
test(L05): add MQTT to VSOA test cases
docs(Z03): complete sensor uplink guide
fix(L03): correct ACK retry handling
chore(common): update submission standard
```

一个提交只处理一个明确任务。不得把三个项目的大量无关修改放在同一个提交中，也不得强制覆盖他人的提交历史。

## 15. Pull Request 要求

Pull Request 应说明：

- 所属项目和实验编号。
- 本次新增或修改的内容。
- 运行环境和验证方法。
- 测试结果及日志、截图位置。
- 未解决问题和可能影响。
- 提交人自检结果。

合并前至少由一名非提交人审核。涉及公共测试框架、协议格式和目录规范的修改，应由三个项目组长共同确认后再形成正式版本。

## 16. 禁止提交内容

- 未经负责人验证的 AI 生成业务代码。
- 密码、Token、密钥、个人隐私和内部敏感地址。
- `.venv/`、`node_modules/`、缓存、临时文件和无关安装包。
- Word 临时文件，例如 `~$文件名.docx`。
- 大量未说明的二进制编译产物或重复压缩包。
- 与实验无关的个人笔记、聊天截图和个人软件配置。
- 无来源的第三方代码、图片、文档或固件。
- 空白测试记录、伪造日志和未经执行的“通过”结果。

## 17. 提交前自检清单

实验负责人提交前逐项确认：

- [ ] 文件已放入正确的项目和实验目录。
- [ ] README 内容完整，命令和路径可用。
- [ ] 正式代码由负责人理解、运行并确认。
- [ ] 环境、依赖、硬件、接线和配置已写清楚。
- [ ] 核心功能能够编译、烧录或运行。
- [ ] 正常、异常和边界场景已有测试。
- [ ] 测试记录包含实际结果、执行人和日期。
- [ ] 日志和截图能够证明实验结果。
- [ ] 配置中没有密码、Token 和个人隐私。
- [ ] 未提交缓存、虚拟环境和无关生成文件。
- [ ] 非作者已按照 README 完成交叉复现。
- [ ] 项目组长已审核，问题已记录或关闭。

## 18. 验收判定

| 结论 | 判定条件 |
| --- | --- |
| 通过 | 必交内容完整，核心功能成功，关键测试通过，证据充分，能够交叉复现 |
| 有条件通过 | 核心功能完成，但存在不影响主要目标的小问题，已记录责任人和修复期限 |
| 待复测 | 已修改问题但尚未完成复测，或硬件、网络条件暂时不具备 |
| 不通过 | 缺少正式代码或关键文档，核心功能失败，测试记录不真实，或无法复现 |

实验只有在代码、文档、测试记录、日志和结果证据均完成后，才能标记为“已完成”。目录已建立、README 已填写或代码已上传，均不能单独作为完成依据。

## 19. 最终交付检查

项目最终提交前，由项目组长汇总确认：

1. 本组所有实验均有明确负责人和验收状态。
2. 项目 README 中的实验列表和仓库实际目录一致。
3. 所有实验均能根据文档复现，关键问题有处理记录。
4. 项目级测试文档、测试记录和验收清单已经更新。
5. 公共协议格式、测试框架和跨组接口已经由相关组长确认。
6. 最终 PPT、实验指导书、代码和测试证据版本一致。
7. GitHub `main` 分支中不存在仅供个人使用的临时内容。

本规范由三个项目组共同执行。后续如需修改目录、公共协议、测试框架或验收条件，应在组长讨论并确认后更新版本，避免各组使用不同标准。
